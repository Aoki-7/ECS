## 发现的问题

### 严重
1. **`World.has_entity()` 不检查 generation**（core/world.py）
   - 只判断 entity_id 是否存在于 EntityManager，不校验传入 Entity 的 generation 是否匹配。过期实体可能被误认为存在。
   - 建议改为 `return self._entity_manager.has_entity(entity)`。

2. **`World.query()` 与 `World.get_components()` 共享 `_query_cache` 但格式不一致**（core/world.py + core/query_api.py）
   - `query()` 缓存格式：`(Entity, comp1, comp2)`（已解包）
   - `get_components()` 缓存格式：`(Entity, [comp1, comp2])`（list）
   - 调用顺序不同会导致返回格式错乱。已用临时脚本验证。
   - 建议：为 Query API 单独缓存，或统一缓存格式。

### 中等
3. **`SystemScheduler` 重复定义 `get` 方法**（core/system_scheduler.py:77-82 与 90-101）
   - 第二个定义覆盖第一个，存在冗余。

4. **`World.update()` 每 tick 全量清除查询缓存**（core/world.py:353）
   - 覆盖了 ArchetypeStore 和 World 的精确缓存失效机制，导致跨 tick 缓存完全失效。

### 低/注意
5. `SystemScheduler._sort()` 每次循环排序整个队列，效率较低。
6. `WorldEventBus.subscribe()` 每次订阅都排序。
7. `WorldEventBus.publish()` 事件过滤失败未计入统计。
8. `World.add_component/remove_component` 发布事件无 timestamp。
9. `World.components` 直接访问 `ArchetypeStore._archetypes`，破坏封装。
10. `PerformanceMonitor` shebang 使用 `python` 而非 `python3`。

## 修复结果（2026-07-13 已完成）

已修复全部 10 个问题并全量测试通过：

1. **`World.has_entity()`**：改为调用 `self._entity_manager.has_entity(entity)`，正确校验 generation。
2. **`World.query()` / `World.get_components()` 缓存冲突**：统一缓存格式为 `(Entity, [components])`，`query()` 迭代时解包，不再冲突。
3. **`SystemScheduler` 重复 `get` 方法**：删除冗余定义。
4. **`World.update()` 每 tick 全量清除缓存**：改为依赖 add/remove 组件/实体时的精确失效，提升缓存复用率。
5. **`SystemScheduler._sort()` 效率**：Kahn 算法队列从每轮排序改为堆（heapq），复杂度从 O(n² log n) 降至 O(n log n)。
6. **`WorldEventBus.subscribe()` 排序**：每次 `sort()` 改为 `bisect.insort_left` 按优先级降序插入。
7. **`WorldEventBus.publish()` 过滤统计**：新增 `filtered` 统计字段。
8. **`World.add_component/remove_component` 事件 timestamp**：与 create/remove_entity 保持一致。
9. **`World.components` 封装**：改为调用 `ArchetypeStore.get_all_components_by_type()`，不再直接访问 `_archetypes`。
10. **`PerformanceMonitor` shebang**：改为 `#!/usr/bin/env python3`。

同步更新的测试：
