## 一、P0 严重 BUG 修复（8 项）

### ✅ BUG-02/03: `seek_target_system.py` 行为常量与字段不匹配

| 项目 | 内容 |
|------|------|
| **文件** | `human/systems/survival/seek_target_system.py` |
| **改动** | `ActionType.MOVE` → `ActionType.MOVE_TO`；`action.target` → `action.target_pos` |
| **原因** | `ActionComponent` 定义的是 `MOVE_TO` 和 `target_pos`，但 `SeekTargetSystem` 写成了 `MOVE` 和 `target`。`MovementSystem` 只读取 `MOVE_TO`，导致实体找到目标后无法移动，原地卡死。 |

### ✅ BUG-06: `money_system.py` buy 返回逻辑反向

| 项目 | 内容 |
|------|------|
| **文件** | `human/systems/economy/money_system.py` |
| **改动** | `return True, 0.0` → `return False, 0.0` |
| **原因** | 扣款失败时返回 `(True, 0.0)`，调用方误认为购买成功。 |

### ✅ BUG-09: 集合迭代时删除实体风险

| 项目 | 内容 |
|------|------|
| **文件** | `eat_system.py` / `drink_system.py` / `gathering_system.py` |
| **改动** | `world.get_components(...)` 结果外加 `list()` 快照 |
| **原因** | 在 `get_components` 循环内部调用 `world.remove_entity()` 删除资源/食物实体，若被删除实体恰好拥有查询组件集中的某个组件，可能触发 `dictionary changed size during iteration`。 |

## 二、P1 短期修复与优化（4 项）

### ✅ BUG-01: `VisionComponent` 野指针

| 项目 | 内容 |
|------|------|
| **文件** | `vision_component.py` / `preception_system.py` / `search_system.py` / `seek_food_system.py` |
| **改动** | `VisionComponent` 新增 `entity_ids` 字段；`PreceptionSystem` 存储 ID 列表；`SearchSystem` / `SeekFoodSystem` 改为实时 `world.query_entity(eid)` 查询 |
| **原因** | 原 `vision.entities` 存储对象引用，实体被 `world.remove_entity()` 销毁后变为"死亡"引用，后续系统迭代时调用 `world.get_component()` 传入死亡实体导致空引用。 |

### ✅ BUG-05: `PlanningSystem` 同 tick 重复规划循环

| 项目 | 内容 |
|------|------|
| **文件** | `human/systems/core/planning_system.py` |
| **改动** | 新增 `_planned_this_tick` 集合，`update` 开头清空，规划后 `add(entity.id)`，重复实体跳过 |
| **原因** | `PlanningSystem` 将动作设为 `IDLE` 后，同 tick 内 `ActionSystem` 又会取新动作，若条件再次触发则产生规划—中断—规划循环。 |

### ✅ PERF-04: 热路径日志降级

| 文件 | 改动 |
|------|------|
| `reproduction_system.py` | `logger.info` → `logger.debug`（2 处） |
| `tribe_system.py` | `logger.info` → `logger.debug`（4 处） |
| **原因** | `INFO` 级别在循环内部高频输出，显著拖慢性能。 |

### ✅ ARC-04: Component 业务逻辑移至 System

| 项目 | 内容 |
|------|------|
| **文件** | `physiology_needs_component.py` / `physiology_needs_system.py` / `sleep_system.py` / `socialize_system.py` |

| **原因** | ECS 原则：Component 只存数据，逻辑放在 System 层。 |

## 五、架构改进总结

| 改进项 | 修复前 | 修复后 |
|--------|--------|--------|
| 系统总数 | 40+ | **50** |
| 生理需求 | `PhysiologyNeedsSystem` 单系统处理 6 种需求 | 拆分为 **5 个独立 System**，单一职责 |
| 天气影响 | 环境数据丰富但未作用于实体 | `WeatherEffectSystem` 直接修改 needs/hp |
| 疾病模拟 | 有 `HealthcareSystem` 但无 `DiseaseComponent`，update 为空 | `DiseaseComponent` + `DiseaseSpreadSystem` + 激活的 `HealthcareSystem` |
| 战斗系统 | `ActionType.ATTACK/DEFEND/FLEE/CHASE` 已定义但无实现 | `CombatSystem` + `CombatAISystem` 完整实现 |
| 对话/冲突 | `DialogueSystem` / `ConflictManagementSystem` 非 System 子类，死代码 | 继承 `System`，接入 ECS 调度，update 有实际逻辑 |
| Vision 引用 | 存储对象引用，存在野指针 | 存储 `entity_ids`，实时查询 |
| 日志级别 | 热路径 `logger.info` 高频输出 | 降级为 `logger.debug` |
| 数值安全 | `HealthComponent.injury` 重复定义导致为 `None` | 恢复默认空字典 |
