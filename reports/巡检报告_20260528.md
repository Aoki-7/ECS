## 【严重问题】

### 🔴 P0-01: `System._update()` 死代码 — 基类 `update()` 未调用 `_update()`

| 项目 | 内容 |
|------|------|
| **风险等级** | 🔴 **P0** |
| **文件路径** | `core/system.py` 第 20-45 行 |
| **问题描述** | `System.update(self, world)` 方法只检查 `self.enabled` 就 return，**从未调用** `self._update(world)`。所有 30+ 个子类均覆盖了 `update()` 而非 `_update()`，导致 `_update()` 方法在整个代码库中无人调用，是纯死代码。 |
| **代码片段** | ```python
def update(self, world: World):
    if not self.enabled:
        return
    # 缺少: self._update(world)  ← 从未执行

def _update(self, world: World):
    raise NotImplementedError(...)  # 死代码
``` |
| **修复建议** | 在 `update()` 末尾添加 `self._update(world)`，或删除 `_update` 并更新文档注释。 |

### 🔴 P0-02: `SimulationLoop` 绕过 `world.systems` 手动调度 — 双调度器架构

| 项目 | 内容 |
|------|------|
| **风险等级** | 🔴 **P0** |
| **文件路径** | `main.py` 第 67-210 行（`_init_systems` + `update`） |
| **问题描述** | `World` 本身有完整的 `systems` 列表和 `update()` 调度器，但 `SimulationLoop` 并未使用 `world.update()`，而是将所有 System 分为 8 组手动遍历调用。这意味着 `world.add_system()` 添加的系统不会被 `SimulationLoop` 执行，形成**双调度器冲突**。 |
| **代码片段** | ```python

``` |
| **修复建议** | 统一调度：所有 System 注册到 `world.add_system()`，仅调用 `world.update(dt)`。 |

## 【架构问题】

### 🏗️ 1. 双调度器架构（严重）

**`SimulationLoop` 不使用 `world.update()`**，而是手动将 System 分为 8 组遍历：
- `self.time_system`（独立 TimeSystem 类）
- `self.env_pipeline`（14 子系统管线）
- `self.human_systems`（19 个系统）
- `self.physiology_systems`（2 个系统）
- `self.biology_systems`（12 个系统）
- `self.rule_systems`
- `self.civilization_system`
- `self._regenerate_resources()`

这意味着 `world.systems` 中仅注册了 `SpaceSystem` 和 `TribeSystem`，其余 45+ 个 System 全部存在 `SimulationLoop` 的局部列表中。新增系统需要修改主循环代码，**违背了 ECS 可插拔的核心原则**。

### 🏗️ 2. `EnvironmentComponent` 过度中心化

**`environment/environment_component.py`**（109 行）集中存放了光、温度、水分、气体、土壤养分、物理扰动等全部环境数据。单一 Component 承载 25+ 字段，违反了"Component 应该是小而专的数据片段"的 ECS 原则。

**建议拆分：**
- `LightComponent`（par, photoperiod, dli）
- `TemperatureComponent`（air_temperature, soil_temperature, day_night_temp_diff）
- `MoistureComponent`（soil_moisture, air_humidity, vpd）
- `SoilComponent`（soil_ph, soil_ec, nitrogen, phosphorus, potassium）
- `WeatherComponent`（wind_speed, rainfall）
- `GasComponent`（co2, o2）

### 🏗️ 3. `SocialSystem.update()` 几乎为空

**`human/systems/social/social_system.py`** 的 `update()` 方法只有阵营默认值设置和三行注释。这是一个"骨架系统"，本该负责的社交互动逻辑（群体归属、声望等）全部缺失。

### 🏗️ 4. Tick 执行链过长

`SimulationLoop.update()` 中，一个时间步需要依次执行：
1. 空间同步
2. 时间推进
3. 环境管线（14 个子系统）
4. 人类系统（19 个子系统）
5. 生理系统（2 个）
6. 生物学系统（12 个）
7. 规则系统
8. 文明系统
9. 资源再生

合计约 **50 个系统串行执行**。任何单个系统的性能问题都会阻塞整个 Tick。

### 🏗️ 5. Event 系统缺失

代码库中虽然有 `EventLogComponent` 和 `EventLog` 用于记录日志，但**没有 ECS 标准的事件总线/Event 机制**。System 之间的通信完全依赖在同一个 Tick 内顺序修改 World 状态，System 间存在隐式耦合。

## 【性能优化建议】

### ⚡ P1: `CompetitionSystem` O(n²) 遍历

**`biology/systems/competition_system.py`** 第 70-71 行：
```python
for i, c1 in enumerate(competitors):
    for c2 in competitors[i + 1 :]:
        self._compete_pair(c1, c2)
```
在实体数量 N 较大时（例如 1000+），O(n²) 的竞争比较将成为瓶颈。建议引入空间哈希或网格索引，只比较相同/相邻格子的实体。

### ⚡ P1: 热路径大量 `print()` 调用

通过搜索发现大量 `print()` 散落在各个 System 的 `update()` 热路径中：
- `civilization_system.py`: 6 处 `print()`
- `construction_system.py`: 1 处
- `technology_system.py`: 1 处
- `death_system.py`: 1 处
- `reproduction_system.py`: 1 处（print `[生育]`）

每个 Tick 50 个 System × 频繁的 `print()` = 显著的 IO 开销。建议替换为 `logging.debug()` 或移至条件日志。

### ⚡ P2: `World.get_component()` 字典查找

每个 System 内部使用 `world.get_component(entity, ComponentType)` 进行字典 `{ComponentType: {entity_id: Component}}` 的双重查找。高频调用路径（如 `EmotionSystem` 在每个实体上调用 7 次不同组件的 `get_component` 或 `get_components`）存在优化空间。建议用 `get_components()` 批量获取，或使用局部引用缓存。

## 【世界模拟缺失维度】

### 1. 🌡️ **温度/气候生理影响**
- 当前 `EnvironmentComponent` 有 `air_temperature`，但实体没有自己的温度模型
- 缺少生物对环境极端温度的应激响应
- **建议：** 引入 `TemperatureComponent`（实体体温） + 温度生理系统

### 2. 🦠 **疾病与流行病**
- 有 `ImmuneSystem` 和 `ImmuneComponent`，但缺少疾病/病原体模型
- 疾病传播机制（空气/接触/水源传播）完全缺失
- **建议：** `DiseaseComponent` + `DiseaseSpreadSystem`

### 3. 🛒 **经济系统**
- `TradeSystem` 仅实现了最简单的交易
- 缺少货币、价值评估、供需关系
- **建议：** `EconomyComponent` + `EconomySystem` + 升级 `TradeSystem`

### 4. 🧠 **高级记忆与遗忘**
- `MemoryComponent` 实现了基本的记忆存储
- 但缺少记忆衰减、遗忘曲线、记忆回溯
- `MemoryComponent.get_event_sentiment()` 有 Bug（不能按时间过滤）
- **建议：** `MemoryDecaySystem`

### 5. 🗺️ **路径规划与导航**
- 当前移动是随机移动（`MovementSystem`）
- 缺少 A* 或类似寻路算法
- **建议：** `PathfindingSystem` + 地图导航数据

### 6. 👥 **群体行为**
- 有 `TribeSystem` 管理部落，但没有群体智能/群体行为
- 羊群效应、人群聚集、恐慌传播缺失
- **建议：** `GroupBehaviorSystem`

### 7. 🎯 **动态任务系统**
- `TaskComponent` 和 `GoalSystem` 存在但任务由需求驱动
- 缺少从外部注入的动态事件（"发现水源"→"搬家"→"建立营地"）
- **建议：** `EventSystem` 事件总线 + `TaskPlanningSystem`

### 8. 🗣️ **语言/交流模拟**
- 有 `SocializeSystem` 但没有实际的语言/信息交换
- 实体间无法传递信息（"那边有食物"）
- **建议：** 信息传递机制 + 知识传播系统

### 9. 🌱 **完整生态链**
- `CompetitionSystem` 处理植物间竞争
- 但缺少捕食者-猎物关系、生态位、食物网
- **建议：** `EcologySystem` + 食物链模型
