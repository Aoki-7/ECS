## 一、严重 BUG 检查（10 项）

### 🔴 BUG-01：实体删除后引用未清理（VisionComponent 野指针）

**影响等级：** 严重  
**文件：** `human/systems/cognitive/preception_system.py` 第 31 行

```python
vision.entities = [world.query_entity(eid) for eid in id_list if eid != entity.id]
```

`PreceptionSystem` 每次更新将 `Entity` **对象引用**存入 `VisionComponent.entities` 列表。当某个实体被 `world.remove_entity()` 销毁后，其 `Entity` 对象的 `generation` 会递增（变为"死亡"态），但 `VisionComponent.entities` 中的引用**不会自动清除**。后续任何系统迭代这个列表并调用 `world.get_component()` 时，传入的 `Entity` 对象 `is_alive()` 返回 `False`，导致空引用。

**连锁影响：** `SearchSystem`（第 88 行 `for candidate in vision.entities`）、`SeekFoodSystem`（第 46 行 `for entity in vision.entities`）均依赖此列表。

**修复建议：**
```python
# 改为存储 entity_id，使用时通过 world.query_entity() 实时查询
vision.entity_ids = list(id_list)
```

### 🔴 BUG-02：Tick 状态污染 — ActionType 常量不匹配

**影响等级：** 严重  
**文件：** `human/systems/survival/seek_target_system.py` 第 65 行

```python
action.current_action = ActionType.MOVE     # ← 错误！应为 MOVE_TO
```

`ActionComponent` 中定义的是 `ActionType.MOVE_TO`，但 `SeekTargetSystem` 写成了 `ActionType.MOVE`。`MovementSystem` 只处理 `ActionType.MOVE_TO`，导致这个系统设置的移动目标**永远不会被执行**，实体在原地卡死。

**对比确认：**  
- `seek_food_system.py` 第 56 行正确使用了 `ActionType.MOVE_TO`  
- `seek_target_system.py` 第 65 行使用了 `ActionType.MOVE`

**修复建议：** `ActionType.MOVE` → `ActionType.MOVE_TO`

### 🔴 BUG-05：无限递归风险 — PlanningSystem 状态循环

**影响等级：** 严重  
**文件：** `human/systems/core/planning_system.py` 第 39–50 行

```python
# 如果正在移动/进食/饮水但生理需求达到危险水平，中断当前动作
if is_critical and action.current_action in (ActionType.MOVE_TO, ActionType.EAT, ActionType.DRINK):
    action.current_action = ActionType.IDLE
    action.status = ActionStatus.IDLE
    action.progress = 0.0
```

`PlanningSystem` 将动作设为 `IDLE` 后，同一个 tick 中 `ActionSystem` 又会尝试从队列取出新动作。如果规划结果再次触发同样的条件，**同一 tick 内可能产生无限循环**。在单线程模型中虽不会真死循环（Python GIL 会限制），但会导致该实体在当前 tick 反复规划—中断—规划。

**修复建议：** 在 PlanningSystem 中添加 `_planned_this_tick` 集合，防止同 tick 重复规划。

### 🔴 BUG-06：MoneySystem 逻辑错误 — buy 返回条件反了

**影响等级：** 中等  
**文件：** `human/systems/economy/money_system.py` 第 61–68 行

```python
def buy(self, seller_id, item, quantity, buyer_id=None):
    ...
    gold_cost = price
    if not self.remove_currency(buyer_id or seller_id, "gold", gold_cost):
        return True, 0.0    # ← 错误！扣款失败应该返回 False
```

当余额不足时 `remove_currency` 返回 `False`，但 `if not ...` 条件进入后返回 `(True, 0.0)`，**调用方会认为购买成功**，实为余额不足。

**修复建议：** `return True, 0.0` → `return False, 0.0`

### 🟡 ARC-02：Query 查询重复

`world.get_components(PhysiologyNeedsComponent, ActionComponent, TaskComponent, SpaceComponent, InventoryComponent)` 这种 **5 组件联合查询** 在多个 Action 子系统中反复出现：

- `eat_system.py` — 5 组件
- `drink_system.py` — 5 组件
- `sleep_system.py` — 3 组件
- `socialize_system.py` — 3 组件

**问题：** 每个 Action 系统都自己查询同样的组件集合，每次遍历 O(N×M)。

**建议：** 通过 ActionComponent.current_action 统一分派，或提取 `ActionQuery` 辅助类。

---

### 🟡 ARC-03：Tick 执行链过长

**main.py** 中注册了 **40+ 个 System**，单 tick 执行链：

```
TimeSystem → SpaceSystem → EnvPipeline(15子系统) → PreceptionSystem → 
EmotionSystem → ThoughtSystem → GoalSystem → IntentSystem → DecisionSystem → 
PlanningSystem → ActionSystem → MovementSystem → SearchSystem → EatSystem → 
DrinkSystem → SleepSystem → PickupSystem → SocializeSystem → 
... → Biology(10+) → Rules(3+) → Civilization
```

单 tick 串行执行 40+ System，其中不少是 O(N) 遍历。300 tick ≈ 12.5 天，**大规模人口下性能堪忧**。

## 三、性能优化

### 🔶 PERF-01：高频对象创建

- **MemoryComponent.add_event()** 每次调用创建一个新 dict（`{"time":..., "type":..., ...}`），而每个实体可能每 tick 记录多次事件。大量小对象高频创建给 GC 带来压力。
- **EventLogComponent.add_event()** 同样每次创建新 dict。

**建议：** 使用 `__slots__` 或 `dataclass` 事件对象，或对象池复用。

---

### 🔶 PERF-02：O(n²) 遍历风险

`TribeSystem.recruit_nearby()` 和 `PairingSystem._find_partner_fast()` 虽然做了优化，但核心逻辑仍然是**双层循环**：

```python
# pairing_system.py 第 58-65 行
for m_info in males:
    for f_info in females:
        ...
```

当人口达到数千时，`O(M×F)` 的配对复杂度不可接受。建议用空间索引优化。

---

### 🔶 PERF-03：重复 Component 获取

`DecisionSystem.update()` 在 `get_components` 中一次获取 7 个组件，然后在 `_make_decision` 中又用 `world.get_component()` 单独获取 3 个额外的组件（Goal、Relation、Age）。这增加了不必要的字典查找。

```python
# 已通过 get_components 获取了 entity
goal = world.get_component(entity, GoalComponent)       # 额外查找
relation = world.get_component(entity, RelationshipComponent)  # 额外查找
age = world.get_component(entity, AgeComponent)         # 额外查找
```

---

### 🔶 PERF-04：热路径日志输出

`ReproductionSystem` 和 `TribeSystem` 在每 tick 的循环内部使用 `logger.info(...)`。INFO 级别即使在生产环境也默认输出，高频 I/O 会导致显著的性能下降。

**文件：** `human/systems/social/reproduction_system.py` 第 68–81 行  
**文件：** `human/systems/social/tribe_system.py` 多处

---

### 🔶 PERF-05：无缓存的计算重复

`HealthSystem` 每 tick 做两次 `world.get_components()` 查询（一次联合查询 + 一次单组件查询），并且用 `processed` set 去重。

```python
# 第一遍：联合查询
for entity, components in world.get_components(HealthComponent, PhysiologyNeedsComponent):
    processed.add(entity.id)
# 第二遍：单组件查询
for entity, components in world.get_components(HealthComponent):
    if entity.id in processed:
        continue
```

这本质上是两次全量遍历。可以合并为一次。
