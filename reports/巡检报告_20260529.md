## 一、严重 BUG 检查（10 项）

### 🔴 BUG-01：ReputationComponent 序列化失效（Component 架构违规）

**影响等级：** 严重  
**文件：** `human/components/society/reputation_component.py` 第 26–49 行

```python
class ReputationComponent(Component):
    def __init__(self, reputation=50.0, social_status="unknown"):
        self._reputation = reputation
        self._specialties: Dict[str, int] = {}
        ...
```

`Component` 基类被 `@dataclass` 装饰，`ReputationComponent` **没有再次添加 `@dataclass`** 且手动定义了 `__init__`。这导致：

- **`asdict(self)` 返回 `{}`**（通过 Python 验证：非 dataclass 子类的 `asdict` 仅序列化父类的字段，而 `Component` 没有字段）
- **`to_dict()` / `from_dict()` 完全失效** — 所有声誉数据无法序列化/反序列化
- `SocialStandingComponent`（同文件第 162 行）和 `FameComponent`（同文件第 180 行）存在相同问题

**修复：** 添加 `@dataclass` 装饰器或用标准 `__init__` 替代手动版本。

### 🔴 BUG-02：HumanDeathSystem 吞掉所有异常

**影响等级：** 严重  
**文件：** `human/systems/physiological/human_death_system.py` 第 81–83 行

```python
try:
    world.remove_entity(entity)
except Exception:
    pass
```

`remove_entity` 过程中可能出现的任何异常（如 `KeyError`、`RuntimeError`）被**完全静默吞掉**。当实体在多个索引/系统中注册时，删除失败会导致**僵尸实体残留**。

**修复建议：** 至少记录日志，或改为只捕获特定的已知异常。

### 🔴 BUG-03：WeatherEffectSystem 未限制 health.hp 上限

**影响等级：** 中等  
**文件：** `human/systems/environment/weather_effect_system.py` 第 59–68 行

```python
health.hp -= heat_damage
health.hp = max(0.0, health.hp)
...
health.hp -= cold_damage
health.hp = max(0.0, health.hp)
```

对 `health.hp` 只做了 `max(0, ...)` 下限限制，**未限制上限**（如 `min(health.max_hp, ...)`）。虽然当前只做减法，但其他 System（如 `SleepSystem`、`HealthSystem`）可以恢复 `hp`，当天气系统与其他系统交替执行时，可能导致 `hp > max_hp`。

**修复建议：** 使用 `health.hp = max(0.0, min(health.max_hp, health.hp - damage))`

### 🔴 BUG-04：SocialNetworkComponent.Community 引用了未定义方法

**影响等级：** 中等  
**文件：** `human/components/society/social_network_component.py` 第 61–64 行

```python
def get_connectivity(self) -> float:
    ...
    for i in range(len(members_list)):
```

### 🔴 BUG-06：世界级方法被多个 System 错误调用

**影响等级：** 中等  
**文件：** `core/world.py` 第 212–215 行

```python
def get_time(self) -> TimeComponent:
```

## 二、ECS 架构问题

### 🟡 ARC-01：System 职责边界清晰度

近期已将 `PhysiologyNeedsSystem` 拆分为 6 个子 System（`HungerSystem`、`ThirstSystem`、`EnergySystem`、`ComfortSystem`、`SocialNeedSystem`、`HealthSystem`），这是正确的方向。

但仍有以下问题：

| System | 问题 | 文件 |
|--------|------|------|
| `WeatherEffectSystem` | 同时处理温度/湿度/风/雨对 6 个指标的影响 | `.../weather_effect_system.py` |
| `TribeSystem` | 更新领地、领袖传承、忠诚度、招募同时做（含大量临时逻辑） | `.../tribe_system.py` |
| `ConflictManagementSystem` | 约 334 行，检测/创建/管理/解决冲突 + 战斗触发 | `.../conflict_management_system.py` |

### 🟡 ARC-02：ReputationComponent 违反 Pure Component 原则

`ReputationComponent` **大量包含业务逻辑**（`adjust_reputation`、`get_status`、`add_knowledge`、`spread_rumor`、`is_honorable`、`get_title` 等）。ECS 原则要求 Component 纯数据，逻辑归 System。

**建议：** 将业务逻辑迁移至 `ReputationSystem`（尚未实现）。

### 🟡 ARC-03：DiseaseComponent 存储逻辑数据

`DiseaseComponent.diseases` 中的每项是 `Dict[str, Any]`（字典），而非 typed dataclass。键名靠字符串约定（`"name"`、`"duration"`、`"damage_rate"`），**无类型检查**。

### 🟡 ARC-04：Query 查询重复

`WeatherEffectSystem` 每 tick 做 `world.get_components(PhysiologyNeedsComponent, HealthComponent, SpaceComponent)` → 全量遍历。而 `HungerSystem`、`ThirstSystem`、`EnergySystem` 等各个子系统又各自做一次 `get_components(PhysiologyNeedsComponent)` — **每个 tick 同一个组件池被遍历 6+ 次**。

### 🟡 ARC-05：Tick 执行链

`main.py` 注册了约 40+ System，执行顺序（按 priority）：

```
TimeSystem(p5) → EventLogSystem(p5) → SpaceSystem(p7) → EnvPipeline(p20) → 
WeatherEffect(p25) → Preception(p30) → Emotion(p30) → Thought(p30) → Goal(p30) →
Intent(p30) → Decision(p30) → Planning(p30) → Action(p30) → Movement(p32) →
Search(p32) → Eat/Drink/Sleep/Pickup/Socialize(p32) → Social(p35) → Pairing(p35) →
Reproduction(p35) → Tribe(p35) → Role(p35) → ConflictMgt(p35) → Dialogue(p35) →
Hunger/Thirst/Energy/Comfort/SocialNeed(p40) → Health(p40) → Death(p42) →
DiseaseSpread(p45) → Combat(p45) → CombatAI(p45) → ... → Biology(50+) → Rules
```

单 tick 串行 40+ System，大规模数据下性能堪忧。

### 🟡 ARC-06：MoneySystem 完全游离于 ECS

`MoneySystem` 不是 `System` 子类，也不是 `Component`。其数据（`balances`、`market_prices`）**无法被 World 管理**，序列化时必须单独处理。

### 🟡 ARC-07：World 的中心化问题

`world.components = defaultdict(dict)` — 所有 Component 集中存储，`get_components` 每次全量筛选。当实体数量达到数万时，CPU 缓存局部性极差（详见 PERF-10）。

## 三、性能优化

### 🔶 PERF-01：组件池重复遍历

每个生理子系统（Hunger/Thirst/Energy/SocialNeed/Comfort）都单独调用 `get_components(PhysiologyNeedsComponent)`，每个 tick 遍历 6 次相同数据。

**建议：** 合并为一个 `PhysiologyUpdateSystem` 内联更新所有生理指标，或使用 `query_components` 缓存。

### 🔶 PERF-02：O(n²) 遍历风险

- `TribeSystem._try_recruit` — 检查所有非部落成员 × 距离
- `PairingSystem` — 双层循环男生 × 女生
- `SocialNetworkComponent.Community.get_connectivity()` — 成员对的 O(m²)

### 🔶 PERF-03：高频小对象创建

`EventLogSystem.add_event()` 每次创建 `dict` 事件对象。当实体数量多时（每 tick 每个实体可能记录 1–3 个事件），GC 压力显著。

**建议：** 限制日志频率（每 N tick 采样），或使用 `__slots__` 事件对象。

### 🔶 PERF-04：热路径日志输出

- `TribeSystem` 多行 `logger.info(...)` 在循环内
- `ReproductionSystem` 第 68–81 行 `logger.info(...)` 
- `HumanDeathSystem` 第 74 行 `logger.info(...)`

在 INFO 级别下这些日志**默认输出**，高频 I/O。

### 🔶 PERF-05：WeatherEffectSystem 未缓存环境组件

每 tick 调用 `world.get_environment()` 并在每个实体循环中访问 `env.air_temperature` 等属性。单 tick 读取数千次，值得局部变量缓存。

### 🔶 PERF-06：HealthSystem 两次遍历

`HealthSystem.update()` 做了两次 `get_components()` 遍历（一次联合 + 一次单组件），用 `processed` set 去重，本质重复遍历。

### 🔶 PERF-07：SpaceSystem 同步仍需每帧全量扫描

`SpaceSystem.update()` 遍历所有 `self.components` 查找 `dirty=True`。对于静态实体（如资源/建筑），大多数从不移动，每次都检查浪费。

### 🔶 PERF-08：Entity ID 回收后的碎片化

`Entity._free_ids` 使用栈式回收（`pop()`）。短期频繁的创建/销毁会导致 ID 分配跳跃，影响哈希表性能。
