# ECS 项目架构迁移方案

## 迁移目标

统一组件版本，消除新旧版本冲突，确保系统一致性和稳定性。

## 迁移范围

### 核心组件
1. **DiseaseComponent**: `biology/components/disease_component.py` → `human/components/health/disease_component_v4.py`
2. **EmotionComponent**: `human/components/cognitive/emotion_component.py` → `human/components/cognitive/emotion_component_v4.py`
3. **MemoryComponent**: `human/components/cognitive/memory_component.py` → `human/components/cognitive/memory_component_v4.py`
4. **SocialComponent**: `human/components/social/social_component.py` → `human/components/social/social_component_v4.py`

### 受影响系统
- `human/entities/human_entity.py` (核心实体模板)
- `human/systems/health/disease_spread_system.py`
- `human/systems/health/healthcare_system.py`
- `presentation/human_observation_system.py`
- `presentation/human_panel.py`
- 所有相关测试文件

## 迁移策略

### 采用"版本重定向 + 兼容接口"策略

1. **保留旧版文件路径**：不删除旧文件，避免大量引用需要修改
2. **修改旧版文件内容**：让旧版文件导出 v4 组件，实现无缝替换
3. **添加兼容方法**：在 v4 组件中添加旧版 API 兼容方法，确保旧代码能正常工作
4. **逐步清理**：迁移完成后，删除不再需要的旧文件

## 实施步骤

### 阶段 1: 备份与评估
- 创建迁移分支或备份关键文件
- 评估所有使用旧版组件的代码
- 列出需要修改的文件清单

### 阶段 2: 组件 API 兼容层
- 在 v4 组件中添加旧版 API 兼容方法
- 修改旧版组件文件，使其转发到 v4 组件
- 确保新旧 API 都能正常工作

### 阶段 3: 核心实体迁移
- 更新 `human/entities/human_entity.py` 使用 v4 组件
- 调整组件初始化逻辑
- 验证实体创建功能

### 阶段 4: 业务系统迁移
- 逐个更新使用旧版组件的系统
- 优先迁移核心系统：disease_spread_system, healthcare_system
- 然后迁移 presentation 层

### 阶段 5: 测试与验证
- 运行全量测试
- 运行演示脚本
- 验证功能正确性

### 阶段 6: 清理与文档
- 删除真正的旧版组件文件
- 清理临时文件
- 更新文档

## 风险缓解

1. **小步快跑**：每次只迁移一个组件，确保稳定后再继续
2. **测试先行**：每个阶段都运行测试
3. **保留回滚**：保留原始文件，必要时可以回滚
4. **兼容层**：通过兼容方法确保旧代码能工作

## 验证标准

- 全量测试 100% 通过
- 演示脚本正常运行
- 无组件版本冲突
- 核心功能正常
