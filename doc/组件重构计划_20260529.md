# Human 组件重构计划 — 通用组件迁移

> **日期：** 2026-05-29  
> **目标：** 将 `human/components/` 下通用组件迁移到更合适的位置，消除 human 目录的"万能垃圾桶"问题

---

## 一、当前问题

`human/components/` 下共有 **45 个组件文件**，涵盖：
- 基础属性（年龄、性别、身体）
- 生理状态（生命值、需求、体温）
- 空间能力（速度、视野、搜索）
- 认知能力（记忆、情绪、思维、性格）
- 社交关系（关系、婚姻、部落、冲突）
- 经济活动（库存、钱包、交易）
- 战斗能力（攻击、防御）
- 伤害状态（出血、骨折、中毒）

**问题：** 动物、植物、建筑等实体也需要部分这些组件，但目前全部放在 human 下，造成：
1. 语义误导 — 非人类实体挂载 "human/components" 下的组件
2. 重复定义风险 — 不同模块可能重复创建类似组件
3. 维护困难 — 找通用组件必须先知道它在 human 下

---

## 三、迁移执行步骤

### Phase 1: 创建新目录结构
```
core/components/           # 新建
biology/components/injury/ # 新建
```

### Phase 2: 移动通用组件（保持向后兼容）
1. 复制文件到新位置
2. 在新文件中更新导入路径
3. 在旧文件中保留 `from 新路径 import XXX` 作为兼容代理
4. 更新所有 System 的导入

### Phase 3: 更新工厂和实体模板
- `HumanFactory` — 更新导入路径
- `AnimalFactory`（未来）— 直接使用 `biology/components/` 下的组件
- `PlantFactory` — 直接使用 `biology/components/` 下的组件

### Phase 4: 清理旧文件
- 确认所有导入已迁移后，删除 `human/components/` 下的代理文件

---

## 四、风险与缓解

| 风险 | 缓解措施 |
|:---|:---|
| 导入路径变更导致大量文件修改 | 保留旧文件为兼容代理，逐步迁移 |
| 循环导入 | 使用 `TYPE_CHECKING` 或延迟导入 |
| 测试覆盖不足 | 每迁移一个组件后运行 `python main.py` 验证 |

---

## 五、验收标准

- [ ] `human/components/` 文件数从 45 降至 20 以下
- [ ] `python main.py` 运行 10 ticks 无异常
- [ ] 非人类实体（如动物）可直接使用 `biology/components/` 下的组件
- [ ] 所有 System 导入路径正确，无 `ModuleNotFoundError`
