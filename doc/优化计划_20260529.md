## 一、P0 严重 BUG 修复（5 项）

| 编号 | 问题 | 文件 | 修复方案 | 难度 |
|:---|:---|:---|:---|:---:|
| **BUG-01** | ReputationComponent 序列化失效 | `human/components/society/reputation_component.py` | 添加 `@dataclass(slots=True)`，删除手动 `__init__`，改为字段声明 | 低 |
| **BUG-02** | HumanDeathSystem 吞掉所有异常 | `human/systems/physiological/human_death_system.py` | `except Exception: pass` → `except Exception as e: logger.warning(...)` | 低 |
| **BUG-03** | WeatherEffectSystem 未限制 hp 上限 | `human/systems/environment/weather_effect_system.py` | `max(0.0, ...)` → `clamp(0.0, max_hp, ...)` | 低 |
| **BUG-04** | SocialNetworkComponent 引用未定义方法 | `human/components/society/social_network_component.py` | 实现 `_get_connection_strength()` 或删除该代码 | 低 |
| **BUG-08** | EventLogSystem 计数器裁剪后未同步 | `core/systems/event_log_system.py` | `_prune_old_events` 中重建 counters | 低 |

### 验收标准
- `ReputationComponent.to_dict()` 返回非空字典
- `HumanDeathSystem` 删除失败时输出 WARN 日志而非静默
- `WeatherEffectSystem` 修改 hp 后始终满足 `0 <= hp <= max_hp`
- `SocialNetworkComponent.Community.get_connectivity()` 不抛 AttributeError
- `EventLogSystem` 裁剪后 `get_event_summary()` 与 `len(events)` 一致

## 二、P1 短期优化（5 项）

| 编号 | 问题 | 文件 | 修复方案 | 难度 |
|:---|:---|:---|:---|:---:|
| **BUG-05** | MoneySystem 完全游离于 ECS | `human/systems/economy/money_system.py` | ① 继承 `System`；② `balances` 拆分为 `WalletComponent`；③ `market_prices` 拆分为 `MarketPricesComponent`（挂载 WorldEntity）；④ 实现 `update()` 动态调价 | 中 |
| **PERF-01** | 生理子系统重复遍历组件池 | `human/systems/physiological/*.py` | 合并为 `PhysiologyUpdateSystem`，一次 `get_components` 内联更新所有指标；或保留独立 System 但共享查询结果 | 中 |
| **PERF-04** | 热路径日志输出 | `human/systems/social/tribe_system.py` 等 | `logger.info` → `logger.debug`（循环内高频输出） | 低 |
| **PERF-05** | WeatherEffectSystem 未缓存环境组件 | `human/systems/environment/weather_effect_system.py` | `env = world.get_environment()` 提到循环外；属性值缓存到局部变量 | 低 |
| **PERF-06** | HealthSystem 两次遍历 | `human/systems/physiological/health_system.py` | 合并为单次遍历，去除 `processed` set 去重 | 低 |

### 验收标准
- `MoneySystem` 被 `world.systems` 管理，`update()` 每 tick 执行
- 生理需求相关 `get_components` 调用从 6 次降至 1 次
- INFO 级别日志不再输出循环内高频消息
- `WeatherEffectSystem` 单 tick `get_environment()` 调用从 N 次降至 1 次
- `HealthSystem` 单 tick 仅遍历一次组件池

## 三、P2 中期架构升级（5 项）

| 编号 | 问题 | 涉及文件 | 方案 | 难度 |
|:---|:---|:---|:---|:---:|
| **P2-1** | 实现 PathfindingSystem | 新建 `human/systems/navigation/pathfinding_system.py` | A* 寻路，接入 `SpaceMap` 障碍物数据；`MovementSystem` 调用路径点而非直线 | 高 |
| **P2-2** | 实现 MemoryDecaySystem | 新建 `human/systems/cognitive/memory_decay_system.py` | 每 tick 扫描 `MemoryComponent`，按时间衰减重要性，低于阈值删除 | 中 |
| **P2-3** | 拆分巨型 System | `weather_effect_system.py` / `tribe_system.py` / `conflict_management_system.py` | 按职责拆分为独立 System（详见下方） | 中 |
| **P2-4** | ReputationComponent 业务逻辑迁移 | `human/components/society/reputation_component.py` → 新建 `human/systems/social/reputation_system.py` | 组件纯数据化，System 接管 `adjust_reputation`/`spread_rumor`/`is_honorable` 等 | 中 |
| **P2-5** | 提取魔法值为常量 | `reproduction_system.py` / `socialize_system.py` / `sleep_system.py` / `weather_effect_system.py` | 模块级常量定义，替换硬编码数值 | 低 |

### P2-3 拆分详情

| 原 System | 拆分为 | 职责 | priority |
|:---|:---|:---|:---:|
| `WeatherEffectSystem` | `HeatEffectSystem` | 高温 >35°C：thirst↑, hp↓ | 25 |
| | `ColdEffectSystem` | 低温 <0°C：energy↓, hp↓ | 25 |
| | `RainEffectSystem` | 降雨 >10mm：comfort↓ | 25 |
| | `WindEffectSystem` | 风速 >10m/s：fatigue↑ | 25 |
| `TribeSystem` | `TerritorySystem` | 领地边界维护 | 35 |
| | `LeadershipSystem` | 领袖传承/选举 | 35 |
| | `LoyaltySystem` | 成员忠诚度变化 | 35 |
| | `RecruitSystem` | 招募新成员 | 35 |
| `ConflictManagementSystem` | `ConflictDetectionSystem` | 检测负面关系，创建冲突 | 31 |
| | `ConflictResolutionSystem` | 冲突解决策略（对话/战斗/和解） | 31 |

## 五、风险与依赖

| 风险 | 影响 | 缓解措施 |
|:---|:---|:---|
| MoneySystem ECS 化改动面大 | 中 | 保留 `MoneySystem` 原有 API 作为 facade，逐步迁移调用方 |
| PathfindingSystem 需 SpaceMap 障碍物数据 | 高 | 先确认 `SpaceMap` 是否已存储地形/建筑障碍 |
| 生理子系统合并可能影响独立优先级 | 中 | 若各子系统需不同 priority，则改为共享查询结果而非合并 System |
| ReputationSystem 迁移后调用方需更新 | 低 | 原 `ReputationComponent` 方法保留为 deprecated 代理 |
